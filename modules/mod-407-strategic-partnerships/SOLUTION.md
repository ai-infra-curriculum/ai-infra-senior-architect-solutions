# SOLUTION — Strategic Partnerships

> Read this *after* you have attempted the partnership
> deliverables. This document explains the senior-architect-tier
> view of partnership strategy.

## What this module is really teaching

Architecture decisions interact with partner relationships:
- Cloud-provider partnerships (preferred pricing, beta access).
- Hardware-vendor partnerships (NVIDIA early access, custom
  silicon).
- Foundation-model partnerships (OpenAI / Anthropic enterprise
  agreements).
- OSS-foundation memberships (CNCF, LF AI).

## What the deliverables should actually look like

### Partnership portfolio (exercise 01)

Current partnerships + their strategic value + cost. Diversified
across cloud / hardware / model / open source.

### Partnership-evaluation framework (exercise 02)

For a candidate partnership: what does it give us, what does it
lock us into, what's the exit cost?

### Vendor-relationship cadence (exercise 03)

Per top-tier vendor: relationship owner, meeting cadence, joint
roadmap reviews. Vendors prioritize customers they actually
talk to.

### Joint-roadmap document (exercise 04)

What we expect from the vendor over 12-24 months; what we
commit to in return.

### Exit planning (exercise 05)

For each partnership: if we needed to leave, how long would it
take? What's the cost? This is the leverage you have at renewal
time.

## Trade-offs we deliberately accepted

- Strategic partnerships are political; they outlive specific
  vendors and people.
- Lock-in is a tax; sometimes worth paying.

## Common mistakes graders see

1. **Vendor concentration without backup plan**.
2. **Partnership without ongoing relationship investment**:
   vendor stops caring.
3. **No exit planning**: leverage evaporates at renewal.
4. **Partnerships made at the engineering level**: don't
   survive when the engineer leaves.

## When to go beyond this implementation

- Move strategic vendor relationships up to the CTO / CIO.
- Build **dual-sourcing** for critical capabilities.
